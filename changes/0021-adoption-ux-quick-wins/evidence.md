# Evidence

## Summary

The four high-priority quick wins from the 0020 product validation are implemented: `adopt` now documents its own adoption Change ("Generated by AIEF during adoption") so it never sits with placeholder evidence; `verify` guides the next step after both PASS and FAIL; `status`/`doctor` no longer raise warnings for optional artifacts (Navigator, Profiles, adapters); and OpenSpec is reported as "not detected (optional)". No new commands, no hidden state, no workflow changes; the adoption Change is documented by the machine but still verified and closed by the human.

## Activities Performed

- Added `adoptionEvidence()` in `cli/src/cli.js`: builds the evidence from what adopt actually did — artifacts are appended to a list at each creation site (AGENTS.md, knowledge/README, profiles/README, each standard, the adoption Change itself) — plus detected signals with reasons, recommended Skills with reasons, and the three guarantees (no functional code changed, no existing files overwritten, only AIEF structure created).
- Updated the adoption tasks.md to reflect reality: machine-done items pre-checked; human items left unchecked (adapt the standards, run verify, close with `aief close --yes --change adopt-aief`). Deliberate design: adopt does not auto-close its Change — "AI assists, humans decide".
- `verify` now ends with a context-aware `Next:`: FAIL → "fix the issues above, then run aief verify again"; PASS with no open Change → honest "no open Change" hint; PASS with an in-progress active Change → prompt-then-close; PASS with a ready active Change → `aief close --yes`. This also makes docs/cli.md's "every command ends with a Next: hint" claim true at last.
- `status` splits required checks (README, AGENTS, Changes — ✓/!) from optional ones (Knowledge, Profiles, Navigator, OpenSpec adapter, Specboot adapter — `· not present (optional)`).
- `doctor` environment reports `· openspec: not detected (optional)` (or `✓ openspec (optional)` when present).

## Verification

- CLI suite: **34/34 pass** (3 new tests: adoption self-evidence content + verify shows ✓ not "in progress"; verify Next on PASS — including the "looks ready" variant — and FAIL; doctor/status optional-awareness with no `!` for Navigator/Profiles/adapters).
- Existing guarantees re-verified by the suite: adopt idempotence, ID-collision safety, app-code-untouched, standards no-overwrite.
- `aief verify` at repo root: PASS (run after closing this Change).

## Findings

- The adoption evidence reuses the exact signal/skill data already computed for the console output — zero extra detection cost, one source of truth.
- Making verify's Next depend on `latestChangeDir()` + `evidenceIsPlaceholder()` needed no new state: the files already answer "what's next".

## Risks

- The "looks ready" hint in verify is heuristic (evidence non-placeholder ≠ evidence good); `close` remains the real gate with its full readiness checks.
