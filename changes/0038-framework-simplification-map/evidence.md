# Evidence

## Summary

The complete classification map of AIEF: every document, command and concept carries exactly one verdict — KEEP / MERGE / ARCHIVE / DELETE — with its replacement, the destination of its information, and its capability check. **Nothing was removed.** ADR-013 recorded.

**Headline: ~150 files → ~45 active. `docs/`: 56 → 10. Commands: 15 → 10. Markdown:code 8:1 → ~3:1 — without deleting a single validated capability.** Every deletion is a duplicate, a corpse, or a second name for behavior that already exists elsewhere.

The map lives in this Change, not as a twelfth file in `docs/aief-2.0/`. Adding a document to a classification map whose purpose is to remove documents would have been the finding, not the deliverable.

## Activities Performed

1. Enumerated every artifact in the repository (~150 files).
2. Traced which templates the CLI actually reads.
3. Ran inbound-reference analysis on every removal candidate (F5's discipline: never archive out from under a live citation).
4. Classified 56 `docs/` files, 25 templates, 15 commands and 15 concepts.
5. Recorded ADR-013 in `knowledge/decisions.md`.

## Risks

| Risk | Severity | Mitigation |
|---|:-:|---|
| **`## Type` → Track executed casually** — it drives prompt composition and the enrichment human-review gate; a careless merge silently drops a governance gate | **High** | Flagged ⚙ as the map's highest-risk item; requires its own design Change. **Not a mechanical merge.** |
| **The map is approved wholesale and executed as one commit** — 53 deletions, 26 merges, no review surface | **High** | Execute per cluster, each independently reviewable and revertible |
| **MERGE degrades into DELETE under time pressure** — the content is "obviously" duplicated, so nobody checks | Med | Every MERGE names its destination; the execution Change must diff content, not just remove files |
| **`propose` / `release` removal breaks an unknown adopter** | Med | Both gated on confirming no adopter depends on them. The only DELETEs that remove behavior |
| Losing the "why" behind an archived evaluation | Low | ARCHIVE, not DELETE, precisely for these |
| **The map is treated as authority** | Med | It is a proposal. Verdicts are a human's to amend, and the DELETE column needs an independent reviewer |

## Recommendations

1. **Decide `## Type` → Track before anything else.** It is the first real bite of ADR-013 and the only classification item that changes CLI behavior. Everything else on the map is documentation.
2. **Execute Stage 2 (deletion) in clusters, not as one commit.** Suggested order — safest first: dead templates (13 files, 0 refs) → `reference-implementation/` (0 refs) → tombstones (3) → `starter-project/` (20) → `docs/navigator/` (22, ⚙ code) → the onboarding merges (11 docs, the highest-value and highest-care work).
3. **Do the `AGENTS.md` unification regardless of everything else.** ADR-004 is violated today; adopted projects receive 8% of AIEF's rules. It is a bug fix, needs no decision, and is the single change most likely to improve real adoptions.
4. **Do not build Tracks yet.** Approved conceptually ≠ approved to build. Stage 1's experiment still gates them, and ADR-013 now also requires that they absorb `## Type` on the way in.
5. **Merge the onboarding cluster last and by hand.** 872 lines collapsing into two files is where information actually gets lost, and it is where the 15-minute criterion is won.

## Lessons Learned

1. **My own inventory was wrong on the first pass** — `docs/navigator/` is 22 files, not 7, because a `*.md` glob does not descend. The audit that criticizes a tool for misreading reality misread reality with the same failure mode: a plausible answer, confidently reported, from a query that could not see everything. Law 5, again, on the author.
2. **"No new documents" and "produce a complete map" are only contradictory if the map must be a document.** It is a spec. AIEF already had a place for it.
3. **The design rule earns its keep immediately.** Nobody would have noticed `Track` colliding with `## Type` while designing Tracks — the two live in different documents. ADR-013 forced the question at classification time, before code.
4. **Duplication clusters around the entry.** Four `AGENTS.md`, three command references, three roadmaps, three responsibility matrices, seven front doors. Nothing duplicates in `cli/src/core/`. **Documentation duplicates where it is read most and verified least.**
5. **Dead things accrete quietly.** 13 dead templates, a 1-line placeholder, three tombstones — none of it was ever decided; it was just never revisited. A framework needs a garbage collector, and ADR-013 is it.
