# Evidence

## Summary

A design study for AIEF 2.0 — eleven deliverables under [`docs/aief-2.0/`](../../docs/aief-2.0/) — produced from a full audit of the AIEF repository and, critically, of the **case-study repository on disk** (`trk-orchestrator-portal`), not only of AIEF's own prior documents about it.

**The finding that reframed the work:** exactly one file in Flux Portal carries a fingerprint of the AIEF CLI — `changes/0001-adopt-aief/`, which AIEF generated about itself. The CLI ran on day 0 and never again. The other 12 Changes and 9,011 lines of governance Markdown were written by hand.

AIEF was adopted as a **folder convention**, not used as a **tool**. F1, F2 and F6 are consequences of that single fact rather than independent defects — a status parser can only recognize the output of a command that never ran.

No implementation. No component deleted. No accepted ADR overridden.

## Activities Performed

1. Audited the AIEF repository: 316 files, 15 commands, 11,602 Markdown lines, ~1,478 production LOC (8:1 ratio).
2. Audited `trk-orchestrator-portal` on disk — artifact-by-artifact, against git history, for what was *touched* rather than *offered*.
3. Read the closure-phase findings (F1–F6), the two prior validations, the external-harness evaluation, the governance conventions and all 12 ADRs.
4. Built the responsibility map and the five-way classification.
5. Designed the experience: one entry point, the six-step flow, Tracks, the eleven-piece harness, the OpenSpec and SpecBoot rules, the roadmap.
6. Dogfooded: created this Change with `aief new-change` and verified it with `aief verify`.

**Scope containment — with a correction.** This Change created exactly two paths: `changes/0037-aief-2-0-experience-redesign/` and `docs/aief-2.0/`. Nothing else was created or modified. No code changed; Change 0036, OpenSpec and SpecBoot untouched.

`git status` is *not* clean, and an earlier draft of this file claimed it was. The working tree already carried **Change 0036's work, uncommitted**, before this Change began — `cli/src/cli.js`, `cli/src/core/`, `cli/templates/ci/`, `docs/dogfooding-findings.md`, `docs/proposals/`, `changes/0036-*/`. Verified by modification time: every one of those was last written between 23:33 and 23:49, while this Change's first artifact was written at 23:58 (`stat -c '%y' <path>`). The session's initial git snapshot reported the tree as clean; it was stale.

Recorded rather than quietly fixed, for two reasons: the accepted-and-validated F1/F2/F3 work is **not committed** and someone should decide whether it should be; and a `git status`-based scope check is exactly the kind of confident-but-wrong signal this study spends eleven documents complaining about — [Law 5](../../docs/aief-2.0/01-vision.md#law-5--when-the-truth-cannot-be-determined-fail-loudly), applied to its own evidence.

## Risks

| Risk | Severity | Mitigation |
|---|:-:|---|
| **AIEF 2.0 ships on top of 1.x** — surface grows, v4 repeats | **High** | Roadmap rule: a stage deletes before it adds; Stage 2 is pure deletion |
| **Law 4 (disproportionate gates get bypassed) is wrong** — the study's load-bearing hypothesis, inferred, not observed | **High** | Stage 1 is the experiment that refutes or confirms it, and it ships before any Track code |
| **n=1** — one project, one team, one deadline | **High** | Every proposal carries an ADR-008 gate; nothing ships on this evidence alone |
| Stage 2 never happens — deleting 40 files demos badly | Med | Named explicitly in the roadmap as the likeliest point of failure |
| `basic` becomes a universal escape hatch | Med | Measure the distribution; if everything is basic, the Track failed |
| F4's field becomes theater — `none` everywhere | Med | Measure the honesty of `none` answers |
| Terminology collision ships (Profile vs Track) | Med | Flagged for human decision before any build |

## Recommendations

1. **Answer first: is AIEF 2.0 a redesign or an addition?** If an addition, stop — the study's premise fails.
2. **Run Stage 1 before building anything.** It costs patience and settles whether Tracks are needed at all. Building the gate and Tracks together makes the result uninterpretable.
3. **Do Stage 2 (deletion) in parallel** — it needs no gate; every item is a duplicate, a tombstone or a self-declared corpse.
4. **Fix `AGENTS.md` regardless.** ADR-004 is violated today; that is a bug, not a redesign, and adopted projects are receiving an 8% version of AIEF's rules.
5. **Ask the Flux authors why `spec.md` stopped at 0008.** One conversation settles what this study can only infer from file listings, and it is the pivot for A9, Law 4 and Stage 5.
6. **Reconsider ADR-012 honestly.** The profile concept has 0 uses across a full migration, and AIEF's prompt promises it anyway.

## Lessons Learned

1. **Audit the adopted project, not the framework's account of it.** Every finding that mattered came from `trk-orchestrator-portal` on disk. AIEF's own documents recorded F1–F6 accurately and still missed A1 — because a framework auditing itself asks what's broken, not what's ignored.
2. **"Nobody ran it" is a design finding, not a discipline problem.** F2 named the fact; framing it as adoption failure hides the question of *why* a correct tool goes unused.
3. **The inverse audit is the requirements document.** What a team hand-builds under deadline (7 ADRs, rollback, parity) is worth more than any feature request. Nobody writes 9,011 lines of governance for fun.
4. **A capability's value is unrelated to its correctness.** `verify` was built, correct, and worthless — it wasn't on the path.
5. **Documentation growth outran capability again**, 8:1, in the repository founded to escape exactly that. The v4 failure mode is not a phase; it is a gradient.
6. **This study is n=1 and says so.** Its central hypothesis is refutable, and the experiment that refutes it is the first roadmap stage. Any recommendation here that survives contact with a second project has earned something; the rest have not.
