# Evidence

## Summary

Bootstrap experience implemented on 2026-07-06. A new user can now run `npm install`, `npm link`, `aief --help`, `aief doctor` and `aief init` from the repository root without the previous `ENOENT: Could not read package.json` failure. The existing CLI in `cli/` was reused; no new CLI structure was created. The spec's `.aief/` layout was rejected in favor of the visible structure (ADR-009 precedence, confirmed by the owner).

## Activities Performed

- Added root `package.json` (`aief-next`, private, `bin.aief -> cli/bin/aief.js`, `npm test` delegates to `cli/`). Root `package-lock.json` generated by `npm install`.
- `cli/src/cli.js`:
  - `--help`/`-h`/`--version`/`-v` handled in `main()`; usage text now shows both `aief init` modes.
  - `doctor`: data-driven `DOCTOR_GROUPS` (Core required / SDD recommended / Build tools optional / Assistants optional), `toolVersion()` numeric extraction (handles `java -version` on stderr), summary line, `exitCode 1` only when required tools are missing.
  - `adopt` body extracted to `runAdoption()`; new `initHere()` reuses it for `aief init` without arguments (detection report + OpenSpec/SpecBoot guidance + numbered next steps). `aief init <name>` unchanged.
  - `COMMAND_HELP` updated for `doctor` and `init`.
- 7 new tests in `cli/tests/cli.test.js` (flags, doctor grouping/required-failure, init current-dir/idempotence/no-`.aief`/code-untouched, init skeleton).
- Docs: new `docs/bootstrap.md`; README "Get the CLI" + adopt section; `docs/cli.md`; `cli/README.md`; CHANGELOG.

## Verification

Commands executed on Linux (Node v22.22.2, npm 10.9.7):

```
cd aief-next
npm install                  # "up to date, audited 1 package" — no ENOENT
npm link                     # "added 1 package" — global aief at ~/.npm-global/bin/aief
aief --version               # aief 0.1.0
aief --help                  # full usage, both init modes listed
aief doctor                  # Core ✓ node/npm/git · SDD ✓ openspec 1.5.0, ⚠ specboot
                             # Build ✓ java 21.0.11/maven/docker, ○ gradle
                             # Assistants ✓ claude, ○ gemini/cursor/codex
                             # Summary: Environment is usable with warnings. (exit 0)
mkdir <tmp>/aief-bootstrap-test && cd <tmp>/aief-bootstrap-test
aief init                    # detection report, AGENTS.md + changes/ + knowledge/ +
                             # profiles/ created, src/app.js byte-identical, no .aief/,
                             # numbered next steps printed
npm test                     # 50 tests, 50 pass, 0 fail (from root and from cli/)
```

Guarantees respected: no functional code changed, nothing overwritten, no hidden state, no assistant coupling (assistants are detected informatively, all optional and equal).

## Findings

- The bootstrap failure was purely a packaging gap: the CLI worked, but only `cli/` was npm-linkable. A root `package.json` pointing at the existing binary closes it with ~15 lines.
- The 0025 spec conflicted with ADR-009 (`.aief/` hidden layout, `.aief/config.json`). Resolved in favor of ADR-009; documented in change.md. ADR-009 untouched.
- `git --version`/`java -version` output needed normalization (numeric extraction) to keep doctor output scannable.

## Risks

- `aief init` without arguments changed behavior: it previously created an `aief-project/` skeleton. The old default was undocumented and unlikely relied upon; both modes are now explicit in help and docs.
- Doctor spawns ~14 `which`/`--version` processes; acceptable (<1s) but worth watching if the tool list grows.

## Recommendations

- Consider publishing to npm (`npx @aief/cli`) as the next bootstrap step.
- SpecBoot detection is marker-based (`specboot`/`.specboot`/CLI); refine when SpecBoot integration lands.

## Lessons Learned

- Accepted ADRs must be checked before implementing an incoming spec: the `.aief/` proposal would have reintroduced exactly what ADR-009 rejected.
- Reusing `runAdoption()` kept `init` at ~25 lines and guaranteed identical safety behavior between `init` and `adopt`.
