# Changelog

## Unreleased

- Change 0026 — Product identity consolidation (documentation only, no runtime changes): README rewritten to answer what AIEF is / why it exists / what it does not do / how it relates to OpenSpec, SpecBoot and assistants / how to install, bootstrap and execute a Change, in under ten minutes; new canonical docs — `docs/VISION.md` (vision, long-term goals, non-goals, design philosophy), `docs/architecture.md` (implemented architecture: Workflow Engine, Prompt Engine, context composition, AGENTS/Profiles/Standards/Skills, evidence, verify, close), `docs/ecosystem.md` (responsibility matrix and integration contracts), `docs/principles.md` (each principle with its why and backing ADRs), `docs/lifecycle.md` (Bootstrap→Adopt→Analyze→Change→Prompt→Implement→Verify→Close with responsible/inputs/outputs per stage); superseded docs (`Vision-and-Principles.md`, `project-lifecycle.md`, `tooling.md`) converted to pointers; pre-CLI era docs (Getting-Started, first-30-minutes, learning-path, mental-model, choosing-your-workflow, navigator guides) updated from "copy starter-project manually" to the implemented CLI flow; roadmap updated to completed status; ADR review: docs aligned with all accepted ADRs, two spec-vs-ADR conflicts resolved in favor of ADRs (SpecBoot artifact ownership per ADR-003/004/010; "Workflow Engine" identity per ADR-001) and the stale `0025-operational-profiles` reference in ADR-012 documented without modifying the ADR.
- Change 0025 — Bootstrap experience: root `package.json` so `npm install` / `npm link` work from the repo root (fixes the ENOENT a new user hit; binary still lives in `cli/bin/aief.js`); `aief --help`/`-h`/`--version`/`-v` flags; `aief doctor` reports the environment grouped by level (Core required: node/npm/git — SDD recommended: openspec/specboot — Build tools optional: java/maven/gradle/docker — Assistants optional: claude/gemini/cursor/codex) with versions and a summary, never failing because of missing optional tools; `aief init` without arguments initializes the *current* directory by reusing the adopt logic (visible structure only, detection report, OpenSpec/SpecBoot guidance, explicit next steps) while `aief init <name>` keeps creating a new skeleton. The spec's original `.aief/` hidden layout was rejected — it conflicted with ADR-009 (no hidden state). New `docs/bootstrap.md`; 8 new CLI tests.
- Change 0024 — Prompt lifecycle guardrails (from the Claude Code validation on trk-orchestrator-portal): re-running `aief prompt` on a Change with real evidence now instructs the assistant to amend, not overwrite (and to report re-verification when nothing changed); the prompt states where results belong (project evidence vs AIEF/tooling feedback); Analysis prompts resolve the "complete only evidence.md" vs close contradiction — evidence is completed or amended, tasks.md stays human-owned unless explicitly delegated, and the assistant reports which tasks appear complete. Derived from files, no hidden state; verify/close untouched.
- Change 0023 — Gemini validation UX fixes: `aief prompt <assistant>` positional syntax (claude/gemini/codex/cursor; `--assistant` wins when both given; unknown values fail with guidance including the `--profile` hint — no more silent Claude fallback); docs and CLI help show the short form as primary; adoption-Change sequence documented (can close before or after the Analysis Change; latest open Change is automatically active) in Workflow.md, README, docs/cli.md and the generated adoption tasks.
- Change 0022 — Visible Skills: `aief adopt` writes the recommended Skills to `knowledge/skills.md` (why recommended, when to use, related standards, prompt context, common risks, evidence expectations; honest "No operational content yet." when applicable; never overwritten); adopt output compacted (no full signal/skill dump — points to doctor and skills.md); `analyze` and `prompt` reference the file when present.
- Change 0021 — Adoption UX quick wins (from the 0020 product validation): `adopt` generates its own adoption evidence ("Generated by AIEF during adoption") so it never sits as placeholder; `verify` prints a context-aware `Next:` after PASS/FAIL; `status`/`doctor` mark Navigator/Profiles/adapters as optional instead of warnings; OpenSpec shown as "not detected (optional)".
- Change 0019 — Workflow clarity (documentation only): canonical three-level workflow in docs/Workflow.md (Context → Feature → Governance) with Mermaid diagram, responsibilities table, with/without OpenSpec guide, `aief close` vs `/archive` comparison and "What AIEF does not do"; all other docs now summarize it; ADR-011. No CLI changes.
- Change 0018 — Standards and Skills as context (Specboot concepts, not copies): `aief adopt` creates editable project standards under `knowledge/standards/` matched to detected signals (never overwriting); `aief analyze` seeds the Analysis Change with detected signals, Skills, standards, inferred risks and open questions; `aief prompt` includes standards and Skill context honestly; Skills carry operational content (promptContext, commonRisks, standardsToRead, evidenceExpectations); OpenSpec adapter documents the real official workflow (propose is an assistant slash command); ADR-010.
- Change 0017 — Close cycle and guided UX: `aief close` runs real readiness checks and `--yes` marks the Change Closed in `change.md` (files remain the only source of truth — no state file); the active Change is the latest open one; `verify` reports in-progress Changes calmly and only warns on closed-but-incomplete evidence; unified `Next:` hints across all commands; README reorganized so integrations appear after the core flow.
- Change 0016 — First real-project validation (Flux Portal): full adoption flow succeeded with zero manual fixes; evidence-driven fixes (Drizzle as postgres signal, `aws-saas-platform` on cognito, doctor output dedup, analyze hint).
- Change 0015 — Public readiness: README rewritten for new users, GitHub Actions CI (CLI + example tests + `aief verify`), `aief prompt --assistant`, full MIT license text, honest evidence capture for Change 0013.
- Change 0014 — Adoption Engine hardening: safe Change IDs in `adopt`, data-driven technology detection with explained Skill recommendations, runtime-validated OpenSpec delegation with loud fallback, complete `help <command>` coverage, CRLF-safe Analysis detection, unified evidence template, 21-test CLI suite, `knowledge/decisions.md`.
