## 3. What the team built because AIEF didn't have it

The inverse audit is more informative than the audit. Under deadline, Flux Portal **invented** — and maintained — exactly what AIEF lacked:

| Invented | Where | Why it mattered |
|---|---|---|
| **7 ADRs** | `knowledge/architecture/decisions/ADR-001..007` | The decisions that outlive the migration. AIEF mentions `adr.md` as an *optional* file inside a Change and offers no template, no location, no command. ADR-007 (n8n tenancy) had to be **reconstructed retroactively** — this is [F4](../dogfooding-findings.md). |
| **Product context** | `knowledge/product/{vision,roadmap,capabilities,use-cases,workstreams}.md` | Why the system exists. AIEF's `knowledge/README.md` says "capture decisions, lessons learned, constraints and project context here" and stops. |
| **Architecture state** | `knowledge/architecture/{current,target,implementation-status}.md` | Where the migration *is*. AIEF has no concept of a system in transition. |
| **Rollback, parity, cutover** | inside `changes/0013` and `evidence.md` files | The migration's actual risk controls. Absent from AIEF entirely. |
| **A status vocabulary** | 3 incompatible hand-rolled formats | AIEF models one bit: closed / not ([F5](../dogfooding-findings.md)). |
| **`tech-debt.md`** | `changes/0013` | Nowhere in the Change model. |

**Read this table as the requirements document for AIEF 2.0.** People do not hand-build artifacts under deadline pressure for fun. Every row is a proven need, at n=1 — which per [ADR-008](../../knowledge/decisions.md) is a signal, not a mandate. The ADR row is the strongest: it was needed, absent, improvised, lost, and then repaired at cost.

## 5. The onboarding maze

The brief asks how a new developer starts. Today, they choose first and learn later.

**Seven doors, no lobby.** `README.md`, `NAVIGATOR.md`, `docs/index.md`, `docs/bootstrap.md`, `docs/Getting-Started.md`, `docs/first-30-minutes.md`, `docs/learning-path.md` — and `docs/index.md` lists **five** of them under a single heading called "Start Here". A heading that offers five starts is not a start.

**Ninety paths, before any code.** [`docs/navigator/decision-tree.md`](../navigator/decision-tree.md) branches on: project exists (2) × OS (3) × tooling combo (3) × assistant (5) = **90 documented paths**. Every one of those questions is asked *before* the user has done anything, and three of the four are irrelevant to what they are trying to do.

**Vocabulary tax.** To read the decision tree you must already know: AIEF, OpenSpec, SpecBoot, Change, ADR, profile, skill, standard, evidence, governance level. The brief's premise — *"do not assume they know ADR, OpenSpec, SpecBoot, governance, dogfooding, Flux Portal"* — is currently violated by the second page.

**Derived time-to-start (~45–60 min):** clone + `npm install && npm link` (5) → read README, 12,145 chars (10–15) → `doctor` (1) → run the Todo example (5) → `init` (1) → choose among 90 paths (5–10) → `new-change` (1) → fill `change.md` + `spec.md` + `tasks.md` **from empty scaffolds** (15–20). *Inference from the documented steps, not a measured user test — [05-user-flow.md §4](05-user-flow.md#4-usability-validation) defines the test that would measure it.*

## 7. What OpenSpec already solves

- Proposal → Spec → Tasks generation (ADR-001/002 already concede this).
- **Contract validation** — the one thing AIEF must never re-implement.
- Contract archival (`/archive`, distinct from `aief close`).

**Overlap to resolve:** `spec.md`. When OpenSpec owns the contract, AIEF's `spec.md` is either a duplicate or a stub. Flux's data is suggestive: `spec.md` was abandoned at Change 0008 while OpenSpec changes kept being written. The correct rule ([08-openspec.md](08-openspec.md)) is that AIEF never holds a contract — it holds a *reference* to one.

## 8. What SpecBoot already solves

- Assistant instruction bootstrap and hierarchy.
- **Modular project standards** — precisely the 6 templates AIEF ships and Flux never edited.
- Role/skill knowledge files.

[ADR-003](../../knowledge/decisions.md) said "concepts, not copies". What happened is subtler and worth naming plainly: AIEF **re-implemented the concept** as its own templates, and they went unused. `docs/VALIDATION-SUMMARY.md` already recorded the smell — *"SpecBoot appeared as copied template residue, not a live standards provider."* The audit confirms it: residue is exactly what it is.

## 9. What genuinely belongs to AIEF

Strip everything OpenSpec, SpecBoot or the assistant does better, and what remains is small, real, and validated:

1. **The Change** — the unit of governed work. 13 of them survived a migration.
2. **Evidence** — 3,655 lines written by hand under deadline. Nobody writes that much of something they don't value. *This is AIEF's validated differentiator*, and both prior validations said so independently.
3. **Closure** — deciding that work is done, with a gate a human must pass.
4. **Orchestration** — knowing what comes next.
5. **Context composition** — assembling the brief ([07-harness.md](07-harness.md)).

Everything else in §4 is a candidate for deletion, delegation, or demotion.
